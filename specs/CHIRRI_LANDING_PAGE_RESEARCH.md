# Chirri Landing Page Research
## What Makes the Best SaaS Landing Pages Convert

*Based on analysis of 20+ landing pages and industry research (March 2026)*

---

## 1. Top 10 Landing Page Patterns

What the best developer tool landing pages have in common:

### Pattern 1: Centered Hero with Product Shot
Nearly every top dev tool (Linear, Vercel, Supabase, Railway) uses a centered hero layout: bold headline in the middle, product visual below. Side-by-side (text left, image right) is the exception, not the rule. Centered = stable, trustworthy.

### Pattern 2: Headline Under 8 Words
- "Your shortcut to everything" (Raycast -- 4 words)
- "Email for developers" (Resend -- 3 words)
- "Build in a weekend, scale to millions" (Supabase -- 7 words)
- "Ship faster" (Neon title tag -- 2 words)
- "Deploy anything without the complexity" (Railway -- 5 words)

The best heroes communicate the product's purpose in one breath.

### Pattern 3: Trust Block Immediately After Hero
Logos or stats appear right after the hero in nearly every page studied. Two flavors:
- **B2B products:** Recognizable company logos (often scrolling carousel)
- **Individual-oriented tools:** Big numbers (GitHub stars, deploy counts, usage stats)

Railway: "46.5M+ deploys per month (and counting)" -- oddly specific = believable.

### Pattern 4: Show the Product, Don't Describe It
Linear shows actual board UI with real-looking issues. Stripe shows code snippets. Vercel shows deploy logs. The product screenshot IS the marketing. Evil Martians' study of 100 dev tools confirms: animated or static product UI in the hero is the dominant pattern.

### Pattern 5: Two CTAs in Hero (Primary + Secondary)
Primary: Bold, action-specific ("Deploy your app", "Download", "Try Chirri")
Secondary: Lighter/outlined ("View docs", "GitHub", "See how it works")

Avoids the "only one path" problem while maintaining visual hierarchy.

### Pattern 6: Features as Problem-Solutions, Not Feature Lists
The weakest pattern is a raw feature list. The strongest is problem-oriented storytelling: surface a pain point, show how the product solves it. Railway does this well: "Instant networking. Zero setup." names the problem (setup hell) and the solution in four words.

### Pattern 7: Social Proof is Curated, Not Auto-Pulled
Nearly all dev tools use hand-picked testimonials styled to look like tweets or GitHub comments. No embeds, no links -- just clean, controlled praise. Even one good quote is better than none.

### Pattern 8: Dark Mode is Dominant but Not Universal
Most dev tools lean dark (Linear, Vercel, Supabase, Neon, Railway, Fly.io). But Stripe and Raycast use light. Better Stack uses light. A/B test data shows dark can outperform light for developer audiences on actual conversions (not just CTR).

**Key insight:** Light pages get higher CTR but dark pages can deliver more total conversions for technical audiences. Chirri's warm light palette is a differentiator -- it stands out in a sea of dark.

### Pattern 9: Minimal Page, Maximum Impact
Most top pages have 5-8 sections total. They're not infinitely scrolling marketing behemoths. The structure that works (per Evil Martians' study of 100 dev tools):
1. Hero
2. Trust block
3. Feature block
4. Social proof
5. Supporting blocks (FAQ, comparison, integrations)
6. Final CTA

### Pattern 10: No Salesy BS
The Evil Martians study puts it bluntly: "No salesy BS. Clever and simple wins." Most pages avoid flashy interactions. Clean design: solid typography, clear layout, breathing room. Developers have finely tuned BS detectors.

## 2. Section Order Blueprint

### Recommended Order for Chirri (with reasoning)

```
1. HERO SECTION
   ├── Eyebrow (optional: "Now in beta" or milestone)
   ├── Headline (3-6 words)
   ├── Subheadline (1 sentence, specific)
   ├── Primary CTA + Secondary CTA
   └── Product visual (animated or static UI)

2. TRUST BLOCK (social proof #1)
   └── Logo carousel OR key metric
   (Reasoning: Establishes credibility immediately.
    Even pre-launch: "Built by [team background]" or beta user count)

3. FEATURE SHOWCASE (the "what it does" section)
   ├── 3-4 core features
   ├── Problem → Solution framing
   └── Product screenshots for each
   (Reasoning: Now that trust is established, explain the product.
    Chess layout or bento grid works best for Chirri's aesthetic)

4. HOW IT WORKS (optional but powerful)
   └── 3-step visual showing setup → monitoring → alerting
   (Reasoning: Reduces perceived complexity.
    Monitoring tools need to show the setup is painless)

5. SOCIAL PROOF #2 (testimonials)
   └── 1-3 curated quotes with names/photos
   (Reasoning: Reinforces features with human validation.
    Even one good quote works for early stage)

6. INTEGRATIONS / COMPATIBILITY (if applicable)
   └── Logo grid of supported platforms/frameworks
   (Reasoning: Developers check "does it work with my stack?"
    This section answers that question at a glance)

7. PRICING (or link to pricing)
   └── Free tier prominent, 2-3 tiers max
   (Reasoning: Developers want to know cost before signing up.
    Most dev tools put pricing on a separate page but link prominently)

8. FAQ (optional, 4-6 questions)
   └── Addresses common objections
   (Reasoning: Reduces friction for on-the-fence visitors.
    Keep it short -- this isn't a docs page)

9. FINAL CTA
   └── Repeat the primary CTA with a closing statement
   (Reasoning: Catch visitors who scrolled the whole page.
    Should feel like a confident close, not desperate)
```

### Why This Order Works

The flow follows a psychological narrative:
1. **Hook** (hero) → What is this?
2. **Validate** (trust) → Who else uses it?
3. **Explain** (features) → What does it do?
4. **Simplify** (how it works) → Is it easy?
5. **Reinforce** (testimonials) → Do people like it?
6. **Check fit** (integrations) → Does it work for me?
7. **Commit** (pricing/CTA) → What does it cost? Let me try it.

This maps to the buyer's journey: Awareness → Interest → Evaluation → Decision.

## 3. Hero Analysis

### What Works

**Ultra-short headlines that state what the product IS:**
| Company | Headline | Words | Why It Works |
|---------|----------|-------|-------------|
| Resend | "Email for developers" | 3 | Instantly clear. Category + audience. |
| Raycast | "Your shortcut to everything" | 4 | Benefit-first, evocative |
| Supabase | "Build in a weekend, scale to millions" | 7 | Concrete timeframe + ambition |
| Stripe | "Financial infrastructure to grow your revenue" | 6 | Authority + specific benefit |
| Linear | "The product development system for teams and agents" | 8 | Clear category definition |
| Neon | "Fast Postgres Databases for Teams and Agents" | 7 | Technical + specific audience |
| Railway | "Deploy anything without the complexity" | 5 | Problem-aware, anti-complexity |
| Fly.io | "Build fast. Run any code fearlessly." | 6 | Two punchy sentences, confident |

**Subheadlines that add one layer of specificity:**
- Raycast: "A collection of powerful productivity tools all within an extendable launcher. Fast, ergonomic and reliable."
- Fly.io: "The platform for devs who just want to ship."
- Stripe: "Accept payments, offer financial services, and implement custom revenue models -- from your first transaction to your billionth."

**Hero visuals that convert:**
1. **Animated product UI** (Linear, Railway) -- shows the tool in action
2. **Static product screenshot** (most common, always valid)
3. **Code snippet** (Stripe, Resend) -- speaks directly to developers
4. **Interactive/live embed** -- power move for narrow-scope tools

### What Doesn't Work

- Headlines longer than 10 words (readers don't finish them)
- Vague benefits without specificity ("Transform your workflow")
- No product visual in hero (abstract art = "what does this even do?")
- Single CTA with no secondary option (loses visitors who aren't ready to sign up)
- Overly clever wordplay that obscures what the product does

### For Chirri

The hero should communicate in under 5 seconds: "Chirri watches your web pages and tells you when they change." The headline should be 3-6 words. The subheadline adds one layer of detail. The visual should show the actual Chirri dashboard or a monitoring flow.

**Hero formula:**
```
[Eyebrow: milestone or status, e.g., "Open beta"]
[Headline: 3-6 words, what Chirri IS]
[Subheadline: 1 sentence, how it helps]
[Primary CTA: "Try Chirri" or "Start watching"]  [Secondary CTA: "See how it works"]
[Product visual: dashboard screenshot or animated monitoring flow]
```

## 4. Feature Showcase Patterns

### Layout Formats (ranked by effectiveness for dev tools)

**1. Chess/Alternating Layout**
Text left + image right, then swap. Used by: Railway, Fly.io, Cronitor.
- Best for: Products with 3-5 distinct features, each needing a visual
- Chirri fit: ★★★★★ (works perfectly for showing different monitoring views)

**2. Bento Grid**
Mixed-size cards in a grid layout. Used by: Linear (implicit), Vercel.
- Best for: Products with many small features or categories
- Chirri fit: ★★★★☆ (matches the bento aesthetic in the design system)

**3. Full Screenshots with Short Descriptions**
Product screenshots with 1-2 sentence descriptions. Used by: Linear.
- Best for: Products where the UI tells the story
- Chirri fit: ★★★★☆ (if the dashboard is beautiful, let it speak)

**4. Tabbed Feature Block**
Features grouped under clickable tabs. Used by: Vercel (Agents/AI/Web/Commerce).
- Best for: Products serving multiple use cases
- Chirri fit: ★★★☆☆ (maybe for different monitoring types)

**5. Step-by-Step**
Numbered steps showing a flow. Common for onboarding/setup explanation.
- Best for: "How it works" sections
- Chirri fit: ★★★★★ (perfect for: 1. Add a URL → 2. Set what to watch → 3. Get chirps)

### Storytelling Approaches (ranked)

1. **Problem-oriented** (BEST for Chirri): "Your competitors changed their pricing page. You found out from a customer tweet." → Show how Chirri catches this.
2. **Action-oriented**: "Watch any page. Detect changes. Get notified." Simple, effective.
3. **Bold statements**: "Every change, caught." Works once Chirri has reputation.
4. **Feature list**: AVOID. Just listing "Visual diff", "API monitoring", "Slack alerts" without context is the weakest approach.

### How Many Features to Highlight

- **3-4 on the landing page** (the core value props)
- **More on a dedicated /features page** (like Linear's /features)
- Rule: Each feature should be immediately understandable from its heading + visual

### For Chirri

Recommended feature showcase:
1. **"Watch anything"** -- show the URL input + configuration UI (the simplicity of setup)
2. **"See what changed"** -- show a visual diff (the core value)
3. **"Know instantly"** -- show notification flow (Slack/email/webhook)
4. **"Track over time"** -- show change history timeline (long-term value)

Use chess layout or bento grid. Every feature gets a real product screenshot.

## 5. Social Proof Strategies

### Pre-Launch / Early Stage (Chirri's current position)

When you don't have thousands of users yet, these work:

1. **Team credibility**: "Built by [relevant background]" -- If the team has relevant experience (monitoring, dev tools, open source), mention it. Neon: "Founded by Postgres committers."
2. **Beta user count**: Even "200+ beta users" is meaningful if specific.
3. **One great testimonial**: One quote from one real user with a real name and photo is worth more than fake social proof.
4. **GitHub stars / open-source signals**: If Chirri has open-source components, show the star count.
5. **Technical credibility signals**: "Checks from 12 regions" or "Sub-second change detection" -- specific technical claims build trust.
6. **"Built with" or "Works with" logos**: Technology logos (Next.js, Tailwind, etc.) signal modernity.

### Post-Launch (for when Chirri grows)

1. **Logo carousel**: 6-12 recognizable company logos, auto-scrolling
2. **Big numbers**: "X pages monitored" / "Y changes detected" / "Z notifications sent"
3. **Curated testimonials**: 3-5 quotes from named users with titles and companies
4. **Contextual proof**: Place quotes next to related features (not just in one block)
5. **Case studies**: "How [Company] uses Chirri to [specific outcome]"

### Where to Place Social Proof

| Position | What | Why |
|----------|------|-----|
| Right after hero | Logos or key stat | Immediate credibility |
| After features | Testimonials | Validates feature claims |
| Before final CTA | Usage number or testimonial | Last push before conversion |

### What to Avoid

- "Trusted by thousands" without evidence (Chirri's copywriting guide explicitly calls this out)
- Generic testimonials ("Great product!" -- useless)
- Auto-embedded tweets (formatting breaks, content may be off-message)
- Fake-looking testimonials (stock photos, no real names)

## 6. Visual Design Trends

### Dark vs Light

**Current landscape (2025-2026):**
- **Dark mode dominant** for dev tools: Linear, Vercel, Supabase, Neon, Railway, Fly.io, OpenStatus
- **Light mode notable exceptions**: Stripe, Raycast, Cal.com, Clerk, Notion
- **Better Stack** uses light mode -- notable because it's a monitoring tool

**The data:**
- Dark pages signal "developer-native" and "modern"
- Light pages signal "accessible" and "professional"
- A/B tests show dark can outperform on conversions for technical audiences, but light outperforms on CTR
- Developer tools using dark mode benefit from screenshots blending naturally into the page

**Chirri's opportunity:** The warm white (Kinari #FAF7F2) palette is a genuine differentiator. In a sea of dark dev tool pages, Chirri's warm light aesthetic stands out. This is Stripe's strategy -- they use white to signal "institutional quality" and "we're different." Chirri can use warm whites to signal "calm, considered, premium."

### Gradients and Color

- **Subtle gradients** are back: Linear uses purple gradients, Vercel uses black-to-dark-grey
- **Flat backgrounds** are still common for the page itself (with gradients in hero sections)
- **Accent colors** are used sparingly -- usually just for CTAs and key UI elements
- **Full-bleed color sections** break up long pages (alternating white/off-white or light/dark sections)

### Animations

- **Subtle wins**: Most top pages use minimal animation -- scroll reveals, gentle fades
- **Heavy animation is rare** and reserved for hero sections (Linear's board animation, Vercel's globe)
- **Scroll-triggered animations** appear on about half of top pages
- **Auto-playing product demos** (like Railway's deploy flow) are effective
- `prefers-reduced-motion` is increasingly respected

**For Chirri:** The petal drift animation is the perfect amount of hero animation. Keep the rest of the page restrained. Scroll-triggered fade-ins for sections are fine. No parallax scrolling.

### Product Screenshots

- **Screenshots are non-negotiable**: The Evil Martians study confirms that showing real product UI is the #1 visual element
- **Dark product screenshots on dark pages** look seamless
- **Light product screenshots on light pages** need clear card borders or elevation
- **Annotated screenshots** (with callout labels) appear on some pages but aren't dominant
- **Browser chrome mockups** (showing the product in a browser frame) are declining -- raw UI is preferred

**For Chirri:** Show the actual Chirri dashboard. With the warm Kinari background and the Gofun white (#FFFFFF) card elevation, product screenshots will naturally stand out. Use the warm-md shadow for elevation.

### Typography Trends

- **Large display headings**: 48-72px for hero headlines (Chirri's text-3xl at 52px is in range)
- **Serif for display, sans for body**: Linear, Stripe, and others use serif fonts for headlines. This matches Chirri's Noto Serif JP / Inter split.
- **Generous line spacing**: 1.2-1.4 for headings, 1.5-1.6 for body
- **Wide letter-spacing on labels**: All-caps labels with tracking-wider is standard

## 8. Anti-Patterns -- What to Avoid

### The Big Sins

1. **Not showing the product**: Abstract illustrations instead of actual UI. Developers want to see what they're getting. If you can't show the product, they assume it's not ready.

2. **Vague headlines**: "Transform the way you work" means nothing. Developers need to know in 3 seconds: what is this, and why should I care?

3. **Feature vomit**: Listing 20 features in a grid. Overwhelming = bouncing. 3-4 core features, well-explained, beats 20 bullet points.

4. **Wall of text**: Long paragraphs kill conversion. The Evil Martians study confirms: dev tools use short text + visuals. If it takes more than one breath, it's too long.

5. **AI slop copy**: "Seamlessly integrate your workflow with our powerful platform." Developers detect this instantly and bounce. (See Chirri's Copywriting Guide for the full list.)

6. **No social proof**: Even one testimonial is better than zero. No social proof = "nobody uses this."

7. **Hidden pricing**: Developers want to know cost before investing time. If pricing is unclear, many will leave.

8. **Slow loading**: Performance IS the first impression for a dev tool. If your landing page is slow, developers will assume your product is too.

9. **Generic stock photography**: Stock photos of "diverse teams collaborating" signal "we're not a real dev tool." Product screenshots > stock photos, always.

10. **Forced sign-up before seeing anything**: Gating all product info behind a sign-up form. Show value first, ask for email second.

### Developer-Specific Bounce Triggers

- **No docs link visible**: Developers look for documentation. If they can't find it, they assume it doesn't exist.
- **No technical details**: "We use AI" without explaining how. Developers want to know the tech.
- **Broken code examples**: If you show code and it has syntax errors, credibility is destroyed.
- **"Enterprise" positioning for indie tools**: If the product is for individual developers, don't dress it up as an enterprise solution.
- **Autoplay video with sound**: Instant bounce.
- **Excessive animation**: Heavy parallax, floating elements everywhere -- distracts from the product.
- **Marketing buzzwords in technical context**: "Leveraging synergies" on a dev tool page is a death sentence.

### Monitoring Tool-Specific Anti-Patterns

- **Not showing what a notification looks like**: The core value of monitoring is knowing when something breaks. Show the notification.
- **Complex setup documentation on the landing page**: Setup should look simple. Save the complexity for docs.
- **No uptime/reliability claims about your own service**: If you monitor other people's uptime, yours better be solid.
- **No pricing clarity**: Monitoring tools are notorious for confusing pricing. Be clear.

## 9. Chirri-Specific Recommendations

### Section-by-Section Blueprint

Based on all research, here's what Chirri's landing page should look like:

---

#### Section 1: Hero
**Background:** Kinari (#FAF7F2) with subtle washi grain texture at 3% opacity
**Layout:** Centered composition

**Elements:**
- **Eyebrow** (optional): Small text like "Open beta" or a milestone, in Noto Serif JP, text-tertiary color

  - Candidates: "Know when the web changes." / "Every change, caught." / "Web change detection for developers."
- **Subheadline**: Inter, text-md (17px), text-secondary. One sentence.
  - Example: "Chirri watches web pages and tells you the moment something changes."
- **CTAs**: Primary (Cha-nezumi button, "Try Chirri") + Secondary (outlined, "See how it works")
- **Product visual**: Animated or static screenshot of the Chirri dashboard showing a detected change, with the sakura chirp dot pulsing. This is where the petal drift animation lives -- 3-5 subtle petals drifting through the hero.

**Reasoning:** The centered layout with serif headline positions Chirri alongside Linear and Stripe in perceived quality. The warm palette differentiates from every dark-mode competitor. The product screenshot immediately shows value. The petal drift adds memorability without being gimmicky.

---

#### Section 2: Trust Block
**Background:** Same Kinari or subtle shift to Shironeri (#F3EEE8)

**Pre-launch version:**
- Key metric: "Watching X pages across Y countries" (when real data exists)
- OR team credibility: "Built by developers who got tired of checking websites manually"
- OR beta testimonial: One real quote from a beta user

**Post-launch version:**
- Logo carousel of companies using Chirri
- Stats: "X,XXX pages monitored • Y changes detected this week • Z ms average detection time"

**Reasoning:** Trust immediately after hero is universal best practice. Even a small signal ("200 developers in beta") helps. The key is specificity -- real numbers, not vague claims.

---

#### Section 3: Feature Showcase
**Background:** Alternating between Kinari and Gofun white cards
**Layout:** Chess/alternating layout (text left + screenshot right, then swap)

**Feature 1: "Watch any page"**
- Show the URL input + configuration UI
- Subtext: Point at any URL. Choose what to watch -- full page, specific elements, or just the text. 30-second setup.

**Feature 2: "See exactly what changed"**
- Show a visual diff view
- Subtext: Visual diffs highlight what's different. No noise. Just the change.

**Feature 3: "Get notified your way"**
- Show notification options (Slack, email, webhook)
- Subtext: Slack, email, webhooks. Chirps arrive where you already work.

**Feature 4: "Track changes over time"**
- Show the change history timeline
- Subtext: Every change, timestamped and archived. Scroll back through the full history.

**Reasoning:** Chess layout provides visual rhythm and ensures each feature gets equal attention. Problem-oriented framing ("Watch any page" implies "you shouldn't have to check manually"). Real product screenshots for each feature -- no illustrations or abstract visuals.

---

#### Section 4: How It Works
**Background:** Gofun white card on Kinari, elevated with warm-md shadow
**Layout:** 3-step horizontal layout

1. **Add a URL** (icon: link/plus) → "Paste any URL and tell Chirri what to watch"
2. **Chirri watches** (icon: eye/bird) → "Checks run on your schedule from multiple regions"
3. **You get a chirp** (icon: bell/petal) → "Changes appear as visual diffs with instant notifications"

**Reasoning:** The "how it works" section reduces perceived complexity. Monitoring tools especially benefit from this -- potential users worry about setup friction. Three steps says "this is simple."

---

#### Section 5: Social Proof
**Background:** Shironeri (#F3EEE8)
**Layout:** 1-3 testimonial cards

- Curated quotes with real names, photos, and titles/companies
- Styled simply -- no fake tweet embeds
- Even one good quote works for early stage

**Reasoning:** After showing features, reinforcing with human validation. Place after features so the quotes feel like confirmation of what was just demonstrated.

---

#### Section 6: Integrations (when applicable)
**Layout:** Grid of integration logos

- Slack, Discord, email, webhooks, API
- Framework logos if there's a code SDK

**Reasoning:** Developers check "does it work with my tools?" This answers that instantly.

---

#### Section 7: Pricing
**Layout:** 2-3 tier cards, free tier on the left and prominent

- Free tier with clear limits
- Paid tier(s) with clear value proposition
- No hidden costs
- Annual/monthly toggle if applicable

**Reasoning:** Developers want to know cost before investing time. Better Stack, Cronitor, and OpenStatus all make pricing transparent. Chirri should too.

---

#### Section 8: FAQ (optional, 4-6 questions)
**Layout:** Accordion-style

Suggested questions:
- "What kinds of changes can Chirri detect?"
- "How often does Chirri check my pages?"
- "Can I monitor pages behind a login?"
- "How is Chirri different from [competitor]?"

**Reasoning:** Addresses common objections without cluttering the main narrative.

---

#### Section 9: Final CTA
**Background:** Full-width section, slightly darker (Shironeri or even dark mode variant)
**Layout:** Centered text + CTA

- Brief closing line: "Your pages are changing. You should know about it."
- Primary CTA button: "Try Chirri free"
- No-pressure subtext: "Free plan. No credit card required."

**Reasoning:** Catches visitors who scrolled through everything. The confident, understated closing matches Chirri's voice.

## 10. Competitive Positioning

### Monitoring Tool Landing Pages vs Dev Tool Landing Pages

| Aspect | Monitoring Tools | Dev Tools | Chirri Opportunity |
|--------|-----------------|-----------|-------------------|
| **Tone** | Professional, enterprise-leaning | Developer-native, casual | Developer-native with warmth |
| **Visual style** | Dashboard-heavy, data-focused | Product UI, clean design | Beautiful product UI |
| **Color** | Often blue/green (status colors) | Dark mode dominant | Warm whites + sakura (unique) |
| **Pricing** | Complex, tiered, feature-gated | Simple, free tier prominent | Simple, generous free tier |
| **Hero** | "Monitor everything" generic | Specific product category | Specific: web change detection |
| **Social proof** | Enterprise logos | Developer names/companies | Developer-focused |
| **Personality** | Low, corporate | Medium to high | High (Kyoto Studio aesthetic) |

### Where Chirri Should Position

**Not here:** Enterprise monitoring (Better Stack, Datadog territory)
**Not here:** Generic "uptime monitoring" (Cronitor, UptimeRobot territory)
**Here:** Web change detection for developers who care about craft

Chirri's positioning is closer to **Resend** (specific tool, developer-native, opinionated) than to **Better Stack** (broad platform, enterprise-focused).

The landing page should feel like:
- **Linear's restraint** (clean, minimal, product-forward)
- **Resend's clarity** (three-word category definition)
- **Stripe's authority** (specific numbers, technical precision)
- **Fly.io's personality** (opinionated, developer-casual)

...filtered through the **Kyoto Studio** aesthetic (warm whites, negative space, sakura signals).

### The Differentiation Stack

1. **Visual differentiation**: Warm light palette in a sea of dark mode
2. **Personality differentiation**: Japanese-inspired design system with real craft
3. **Product differentiation**: Web change detection, not generic monitoring
4. **Voice differentiation**: Technical, understated, specific (per Copywriting Guide)
5. **Interaction differentiation**: Sakura petal moments (memorable, unique)

| Cronitor | Cron + uptime monitoring | Different category entirely |
| OpenStatus | Open-source status pages | Complementary, not competitive |
| StatusPage | Atlassian enterprise status | Different market segment |

Chirri doesn't compete with monitoring platforms. It creates a new category: **web change detection as a developer tool**. The landing page should define this category clearly and own it.
